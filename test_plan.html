Test Plan

Introducing and creating a test plan in the early stage of the application will provide the guarantee that we achieved quality when we launch the project with no or minimal bugs. Delivery Drone project.

The system has been designed using modular architecture. A modular architecture means assembling a collection of smaller functional blocks, which makes for a system assembled with a specific purpose in mind. One of the advantages of a modular architecture for test systems is the ability to custom-build the system to fit your exact measurement criteria, which is harder to do with traditional benchtop instrumentation.

In the modularization test technique test cases are broken down into logical, manageable functions or modules. These modules are isolated to create independent tests and they can later be reused or combined with other tests to create complex in-depth test scenarios. Test cases can be referenced in other test cases to create nested reusable structures. As a result, there's less maintenance effort because by updating the corresponding module all test cases reusing it are also updated in one go.

For the test plan, crucial modules will be implemented and tested first. Modules will be tested independently and the system will be tested after the modules are integrated. Some functionality can only be tested by the collaboration and parallel execution of different modules integrated together.

Benefits of this approach will be that:

1. It is easier to identity errors in independent modules as it will be easier to manage data flows and the testing tasks will be smaller
2. Some functionalities of modules will be dependent on the interaction between modules and this can be tested after the modules are integrated into the system
3. Since initially modules can will be developed and tested independently, we can work parallely on the development of modules to speed up development
4. Reusability

We will conduct our tests in four phases.

Phase 1 - Control Module
In this phase we will conduct testing on the control module and it's functionality which can be tested independently.

Phase 2 - Feedback Module
In this phase we will conduct testing on the feedback module and it's functionality which can be tested independently.

Phase 3 - Hardware
In this phase we will conduct testing on the hardware for the system and make sure that the hardware components all work as expected individually. One might not consider this as “software testing” but this is an important step before integrating the control module, feedback module and the hardware. Once we can verify that the hardware is functioning then if there were hardware issues or errors once it was integrated with the other modules, we will know that the likely cause is because of the integration/software. If this step is avoided, a cause for a hardware error could be hardware malfunction and the team might waste time thinking that the problem was software related.

Phase 4 - Integrating the Control, Feedback and Hardware modules
In this phase we will integrate the control module, the feedback module and the hardware module. We will test the functionalities of each module which depends on the collaboration and exchange of information between the modules. The system will be tested to make sure that it is functioning as intended and that the integration was successful.
